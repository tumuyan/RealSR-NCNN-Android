## 1. 通用参数说明

以下参数在**所有6个程序**中都支持：

| 参数   | 全称          | 类型     | 默认值      | 说明                      |
| ---- | ----------- | ------ | -------- | ----------------------- |
| `-i` | input-path  | 路径（必填） | 无        | 输入图片路径或目录（递归扫描）         |
| `-o` | output-path | 路径（必填） | 无        | 输出图片路径或目录               |
| `-h` | help        | 标志     | -        | 显示帮助信息                  |
| `-v` | verbose     | 标志     | 关闭       | 详细输出模式                  |
| `-f` | format      | 字符串    | 空        | **强制输出格式**（忽略alpha检测）   |
| `-e` | format      | 字符串    | 空        | **建议输出格式**（自动转换alpha）   |
| `-k` | skip-size   | 整数     | 0        | 跳过已存在且大小≥阈值的文件（字节，0=禁用） |
| `-p` | pattern     | 字符串    | `{name}` | 批量模式下的文件命名模板            |

### 支持的文件格式

**输入格式**：jpg、jpeg、png、bmp、webp、tif、tiff\
**输出格式**：png、jpg、jpeg、webp、bmp、tif、tiff

## 3. 输出路径与文件名规则

### 3.1 核心规则

输出文件的生成遵循以下**优先级顺序**：

```
用户指定格式 (-f/-e) > 输出路径扩展名 > 输入文件扩展名
```

### 3.2 文件名生成逻辑

#### **场景A：输入为文件，输出为目录**

```bash
realsr-ncnn -i /input/photo.jpg -o /output/
```

**输出结果**：`/output/photo.jpg`（保持原文件名）

**生成规则**：

1. 提取输入文件名：`photo.jpg`
2. 去掉扩展名：`photo`
3. 确定扩展名（按优先级）：
   - 如果使用了 `-f` 或 `-e` → 使用指定的扩展名
   - 否则 → 保持原扩展名 `.jpg`
4. 拼接完整路径：`/output/` + `photo` + `.jpg`

***

#### **场景B：输入为文件，输出为文件**

**情况1：输出路径带扩展名**

```bash
realsr-ncnn -i /input/photo.png -o /output/result.webp
```

**输出结果**：`/output/result.webp`（完全使用输出路径）

**情况2：输出路径不带扩展名**

```bash
realsr-ncnn -i /input/photo.jpg -o /output/result
```

**输出结果**：`/output/result.jpg`（自动补充扩展名）

**生成规则**：

- 有扩展名 → 直接使用该路径作为最终输出
- 无扩展名 → 根据 `-f/-e` 参数或输入扩展名自动补充

***

#### **场景C：输入为目录，输出为目录（批量处理）**

```bash
waifu2x-ncnn -i /photos/ -o /enhanced/ -e webp
```

**假设** **`/photos/`** **目录结构**：

```
/photos/
├── vacation/
│   ├── IMG001.jpg
│   └── IMG002.png
└── portrait/
    └── selfie.bmp
```

**输出结构**（保持目录层级）：

```
/enhanced/
├── vacation/
│   ├── IMG001.webp    ← 使用 -e 指定的格式
│   └── IMG002.webp
└── portrait/
    └── selfie.webp
```

**生成规则**：

1. **递归扫描**输入目录及子目录中的所有支持格式的图片
2. **保持相对路径结构**映射到输出目录
3. 文件名可通过 `-p` 参数自定义模板

***

#### **场景D：输入为目录，输出为文件路径（特殊行为）**

🔍 **这是一个有趣且容易被误解的场景！**

**实际行为**：程序会将输出"文件路径"**当作目录名来创建**，所有图片都会被正常处理并保存到该目录中！

```bash
realsr-ncnn -i /photos/ -o result.jpg
#         ^^^^^^^^   ^^^^^^^^^^
#         输入是目录  输出看起来像文件
```

**执行后的文件结构**：

```
当前目录/
└── result.jpg/          ← 这是一个文件夹！📁（不是图片文件）
    ├── vacation.jpg     ← 处理后的图片1
    └── selfie.png      ← 处理后的图片2
```

**日志输出示例**：

```
/photos/vacation.jpg -> result.jpg/vacation.jpg done
/photos/selfie.png -> result.jpg/selfie.png done
```

⚠️ **重要发现**：

- ✅ 每张图片都被完整处理，没有遗漏
- ✅ 每张图片保存到独立的文件，没有覆盖
- ⚠️ 程序会创建一个名字看起来像文件的**文件夹**

**详细技术分析见[第9章：程序内部机制解析](#9-程序内部机制解析)**

## 4. 输入/输出的4种组合场景

### 场景矩阵表

| 输入类型 \ 输出类型 | 输出为**文件**路径  | 输出为**目录**    |
| ----------- | ------------ | ------------ |
| **输入为文件**   | ✅ 单张处理       | ✅ 单张处理（自动命名） |
| **输入为目录**   | ⚠️ 特殊行为（见下方） | ✅ 批量处理（推荐）   |

> 💡 **提示**：当输入为目录时，即使输出看起来像文件路径，程序也会将其作为目录处理。详细说明见[组合4](#组合4目录--文件路径特殊行为)。

***

### 详细说明

#### 组合1：文件 → 文件

```bash
realsr-ncnn -i photo.jpg -o result.png
```

**行为**：

- 处理单张图片
- **强制保存到** `result.png`
- 忽略任何格式参数（以实际输出路径为准）

**适用场景**：精确控制输出位置和名称

***

#### 组合2：文件 → 目录 ⭐ 推荐

```bash
waifu2x-ncnn -i image.png -o ./output/ -e jpg
```

**行为**：

- 处理单张图片
- **自动命名为** `./output/image.jpg`
- 应用格式参数（`-e jpg`）

**输出路径公式**：

```
{输出目录}/{原文件名}.{格式参数或原扩展名}
```

**适用场景**：需要保留原文件名但改变格式/位置

***

#### 组合3：目录 → 目录 ⭐⭐ 最常用

```bash
srmd-ncnn -i /raw_photos/ -o /processed/ -e png -v -k 10000
```

**行为**：

- **递归扫描** `/raw_photos/` 及其所有子目录
- 处理所有支持的图片格式（jpg/png/bmp/webp/tiff）
- **保持目录结构**映射到 `/processed/`
- 应用格式参数（`-e png`）
- 启用详细日志（`-v`）
- 跳过已存在且≥10KB的文件（`-k 10000`）

**输出示例**：

```
输入: /raw_photos/2024/trip/IMG001.jpg
输出: /processed/2024/trip/IMG001.png
```

**适用场景**：批量处理大量图片

***

#### 组合4：目录 → 文件路径 ⚠️ 特殊行为

```bash
realcugan-ncnn -i /images/ -o final_result.jpg
```

**实际行为**（经过源码验证）：

1. 程序检测到输入是目录，进入批量处理模式
2. **忽略输出的"文件外观"**，直接将其当作目录名
3. 创建名为 `final_result.jpg` 的文件夹
4. 递归处理 `/images/` 中所有图片
5. 每张图片独立保存到该文件夹中

**执行后的结构**：

```
当前目录/
└── final_result.jpg/     ← 这是一个文件夹！📁
    ├── photo1.png       ← 处理完成
    ├── photo2.png       ← 处理完成
    └── photo3.png       ← 处理完成
```

**关键代码逻辑** ([image\_processor.h](common/image_processor.h) 第286-302行)：

```cpp
if (input_is_dir)
{
    path_t output_dir = outputpath;  // 直接将输出路径当目录用
    
    bool output_exists = path_is_directory(outputpath);
    if (!output_exists)
    {
        // 强制创建目录，不管名字是否像文件！
        create_directory_recursive(output_dir);
    }
    
    // ... 继续批量处理所有图片
}
```

**特殊情况处理**：

| 场景              | 行为     | 结果     |
| --------------- | ------ | ------ |
| 输出路径不存在         | 创建为目录  | ✅ 正常工作 |
| 输出路径已存在且是目录     | 直接使用   | ✅ 正常工作 |
| 输出路径已存在且是**文件** | 尝试创建失败 | ❌ 报错退出 |

**错误示例**：

```
# 如果当前目录已经有一个叫 final_result.jpg 的图片文件
realcugan-ncnn -i /images/ -o final_result.jpg

# 错误输出：
failed to create output directory final_result.jpg
```

**适用场景**：一般不推荐使用，除非明确了解此行为。建议始终使用明确的目录路径。

## 5. 格式控制参数优先级

### 5.1 三层优先级体系

```
优先级从高到低：

1️⃣ -f （强制格式）        ← 最高优先级
   ↓ 未指定时降级到 ↓
2️⃣ -e （建议格式）        ← 中等优先级
   ↓ 未指定时降级到 ↓
3️⃣ 原始输入文件扩展名     ← 默认行为
```

### 5.2 代码实现逻辑

```cpp
// 所有程序统一的核心代码
path_t effective_format = format.empty() ? suggested_format : format;
int ret = collect_input_output_files(inputpath, outputpath, effective_format, ...);
```

**解释**：

- 如果 `format`（即 `-f`）不为空 → 使用 `-f` 的值
- 否则如果 `suggested_format`（即 `-e`）不为空 → 使用 `-e` 的值
- 都为空 → 使用输入文件的原扩展名

### 5.3 典型用法对比

| 命令示例                                      | 行为         | Alpha处理      |
| ----------------------------------------- | ---------- | ------------ |
| `realsr -i in.png -o out/`                | 保持原格式      | ✅ 自动转PNG     |
| `realsr -i in.png -o out/ -e jpg`         | 输出JPG（建议）  | ✅ 有alpha转PNG |
| `realsr -i in.png -o out/ -f jpg`         | **强制JPG**  | ❌ 忽略alpha    |
| `realsr -i in.png -o out/ -e png -f webp` | **强制WEBP** | ❌ 忽略alpha    |

## 6. Alpha通道处理逻辑

### 6.1 触发条件

当满足**以下两个条件**时，程序会进行Alpha通道检测和处理：

1. ✅ 输入图片包含透明通道（RGBA/4通道）
2. ✅ **未使用** `-f` 强制格式参数（即 `format.empty()` 为真）

### 6.2 处理流程

```
读取图片 → 检测到Alpha通道？
                ↓
         ┌─────┴─────┐
        是           否
         ↓             ↓
  目标格式是JPG?    正常输出
    ↓ 是              ↓
  自动改为PNG      完成
```

### 6.3 示例演示

**命令**：

```bash
waifu2x-ncnn -i transparent.png -o output/ -e jpg
```

**执行过程**：

1. 读取 `transparent.png`（检测到有Alpha通道）
2. 用户指定了 `-e jpg`（建议格式）
3. 因为**未使用** `-f`，触发Alpha检测
4. 检测到目标格式 `jpg` 不支持透明度
5. **自动转换为** PNG格式
6. 最终输出：`output/transparent.png`

**日志输出**：

```
image transparent.png has alpha channel ! transparent.png will output transparent.png
```

**如果改用强制格式**：

```bash
waifu2x-ncnn -i transparent.png -o output/ -f jpg
```

→ 输出：`output/transparent.jpg`（**丢弃Alpha通道**）

## 7. 批量处理跳过机制

### 7.1 功能说明

通过 `-k` 参数可以跳过已经处理过的文件，避免重复计算。

### 7.2 工作原理

```bash
-k <size_threshold>
```

**判断逻辑**：

1. 计算输出文件的**绝对路径**
2. 检查该路径是否**已存在文件**
3. 如果存在，检查**文件大小是否 ≥ 阈值**
4. 如果都满足 → **跳过该文件**，继续下一个
5. 如果任一条件不满足 → **正常处理**

### 7.3 使用示例

#### 示例1：基本跳过

```bash
realsr-ncnn -i photos/ -o enhanced/ -k 10240
```

- 跳过所有已存在的且大小 **≥ 10KB** 的输出文件
- 小于10KB的文件（可能是损坏的空文件）会被重新处理

#### 示例2：配合详细日志

```bash

```

**输出日志示例**：

```

[skip] batch/photo002.png -> result/photo002.webp (exists, size=98200 bytes >= threshold)
Processing batch/photo003.png...
Processing batch/photo004.png...
```

#### 示例3：中断恢复场景

```bash

```

**阈值设置建议**：

| 场景      | 推荐阈值           | 说明         |
| ------- | -------------- | ---------- |
| 一般图片    | `1024` (1KB)   | 跳过空文件或错误文件 |
| 高质量PNG  | `10000` (10KB) | 确保文件完整性    |
| 大尺寸WebP | `50000` (50KB) | 确保压缩完成     |
| 禁用跳过    | `0` 或不写 `-k`   | 处理所有文件     |

## 🔧 故障排除

### 常见问题

**Q1：为什么输出格式不是我想的？**

- 检查是否使用了 `-f`（强制）还是 `-e`（建议）
- 如果图片有Alpha通道且使用了 `-e jpg`，会自动转为PNG
- 解决方案：使用 `-f jpg` 强制输出JPG（但会丢失透明度）

**Q2：批量处理时某些文件被跳过？**

- 检查是否使用了 `-k` 参数
- 查看 `-v` 日志确认跳过原因
- 解决方案：移除 `-k` 或增大阈值

**Q3：内存不足怎么办？**

- 减小 `-t` 参数（tile size）
- 从 `0`（自动）改为具体数值如 `128` 或 `64`
- 减少并行线程数 `-j`

**Q4：如何提高处理速度？**

- 使用多GPU：`-g 0,1,2`
- 增加处理线程：`-j 1:4:4`
- 减小tile size：`-t 128`（减少内存占用）
- 使用Resize代替AI模型（仅需缩放时）

### 9.1 程序如何判断输出是目录还是文件？

#### 判断函数实现

定义位置：[filesystem\_utils.h](common/filesystem_utils.h) 第32-36行（Windows）/ 第64-70行（Linux）

**Windows版本**：

```cpp
static bool path_is_directory(const path_t& path)
{
    DWORD attr = GetFileAttributesW(path.c_str());
    return (attr != INVALID_FILE_ATTRIBUTES) && (attr & FILE_ATTRIBUTE_DIRECTORY);
}
```

**Linux/Android版本**：

```cpp
static bool path_is_directory(const path_t& path)
{
    struct stat s;
    if (stat(path.c_str(), &s) != 0)
        return false;  // 文件不存在 → 返回 false（不是目录）
    return S_ISDIR(s.mode);  // 存在 → 检查是否是目录属性
}
```

**关键点**：

- Windows: 使用 `GetFileAttributesW` API + 检查 `FILE_ATTRIBUTE_DIRECTORY` 标志
- Linux: 使用 `stat()` 系统调用 + 检查 `S_ISDIR` 宏
- **如果路径不存在**：两个平台都返回 `false`（不是目录）

### 9.3 为什么"目录→文件路径"会创建文件夹？

这是**设计选择而非bug**。核心原因在 [image\_processor.h](common/image_processor.h) 第286-302行：

```cpp
bool input_is_dir = path_is_directory(inputpath);

if (input_is_dir)  // 只要输入是目录
{
    path_t output_dir = outputpath;  // ← 直接赋值，不做任何检查！
    
    bool output_exists = path_is_directory(outputpath);
    // ↑ 只检查"是否存在且是目录"
    // ↑ 不检查"看起来像文件还是目录"
    
    if (!output_exists)
    {
        create_directory_recursive(output_dir);
        // ↑ 强制创建！不管名字是什么！
    }
    
    // ... 后续正常批量处理
}
```

**设计理念**：

1. **简化用户操作**：不需要先手动创建输出目录
2. **统一行为**：输入是目录时，输出一定也是目录结构
3. **灵活性**：允许任何合法的目录名（即使包含`.`或看起来像文件）

### 9.5 主循环处理机制

```cpp
const int count = ltp->input_files.size();  // 图片总数

#pragma omp parallel for schedule(static, 1) num_threads(ltp->jobs_load)
for (int i = 0; i < count; i++) {
    const path_t &imagepath = ltp->input_files[i];     // 各自独立的输入
    // ...
    v.outpath = ltp->output_files[i];                   // 各自独立的输出

    imwrite(v.outpath, image);  // 保存到不同位置！
}
```

**关键发现**：

- ✅ 使用 OpenMP 并行循环遍历所有图片
- ✅ 每张图片有**独立的**输入和输出路径
- ✅ 所有图片都会被处理和保存
- ✅ **没有任何覆盖行为**（除非输出路径恰好相同）
